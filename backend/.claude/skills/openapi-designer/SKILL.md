# OpenAPI Spec Generation — RpgAssistant

This project follows an **API-first** workflow. The OpenAPI document at
`backend/Utilities/Contract.yaml` is the single source of truth for every
HTTP contract. Code follows the contract — never the other way around.

## When to use this skill

- Adding or changing an endpoint.
- Adding or changing a request / response schema or query/header parameter.
- Adding or renaming an error response.
- Regenerating the frontend NSwag client after a contract change.
- Reviewing the contract against the actual API implementation.

## Source of truth

| File | Role |
|---|---|
| `backend/Utilities/Contract.yaml` | **Source of truth.** Edit this first. |
| `backend/src/RpgAssistant.Api/Endpoints/*Endpoints.cs` | Minimal-API endpoints — must conform to the contract. |
| `backend/src/RpgAssistant.Api/Dtos/*Dto.cs` | DTOs — must mirror the schemas in the contract. |
| `frontend/src/services/httpClients/RpgAssistantClient.ts` | **Generated by NSwag from `Contract.yaml`.** Never hand-edit. |

## API-first workflow

For every contract-affecting change, follow this order:

1. **Edit `backend/Utilities/Contract.yaml`** — add or modify the path,
   parameter, schema, or response. Validate the file (see "Validation"
   below) before continuing.
2. **Add an example for every request body and every non-empty success
   response** (see "Examples" below). Examples must satisfy the schema
   constraints.
3. **Regenerate the frontend client** (`RpgAssistantClient.ts`) via the
   NSwag toolchain, then update `frontend/src/services/RpgAssistantService.ts`
   to expose the new operation with friendly types.
4. **Implement the backend**: DTO record under `Api/Dtos/`, command/query
   under `Application/`, endpoint mapping in `Api/Endpoints/*Endpoints.cs`,
   exception → status mapping in `ResultsToHttpResponses` if a new domain
   exception is introduced.
5. **Write tests** that exercise both the happy path and at least one
   not-happy path (validation, not-found, precondition).

If you discover the contract drifts from the implementation, fix the
**contract first**, then bring the code back into line.

## Mandatory schema constraints

Every schema in `Contract.yaml` MUST declare explicit bounds. Reviewers
should reject schemas that leave them implicit.

| Property type | Required keywords | Notes |
|---|---|---|
| `string` (free text) | `minLength`, `maxLength` | Both are required, even if `minLength: 0`. |
| `string` with `format: uuid` / `date-time` / `date` / `email` | format only | Length is implied by the format; do not add min/max. |
| `string` with a regex | `pattern` + `minLength`/`maxLength` | Pattern alone is not enough — set explicit bounds too. |
| `integer`, `number` | `minimum`, `maximum` | Both required. Use `exclusiveMinimum`/`exclusiveMaximum` when needed. |
| `array` | `minItems`, `maxItems` + bounded `items` | The element schema must itself follow these rules. |
| `object` | required keys via `required: [...]` | Every non-optional property listed; no orphan optionals. |
| Any property | `description` | Required if the name is not self-explanatory. |
| Any nullable property | `nullable: true` | Be explicit — never rely on absence. |

When the upper bound is genuinely unknown, pick the smallest defensible
value (e.g. names ≤ 50, descriptions ≤ 500, page sizes ≤ 100, arrays of
ids ≤ 50). Bounds are part of the contract — write them down, do not push
the choice onto the consumer.

### Project-specific defaults

These already exist in `Contract.yaml` — reuse them, do not invent
alternatives:

- Character names (`CharacterDto.name`, `CreateCharacterDto.name`,
  `UpdateCharacterDto.name`): `minLength: 1`, `maxLength: 50`.
- Free-text descriptions (`CreateKnowsDto.description`): `minLength: 0`,
  `maxLength: 50`.
- Filter strings (`NameFilter`, `SortType`): `minLength: 0`, `maxLength: 50`.
- Id arrays (`CharacterPayload.knowCharacterIds`): `minItems: 0`,
  `maxItems: 50`, items `format: uuid`.
- Page/size integers: `minimum: 0` (raise to `1` for genuine 1-based pages).
- Always model error responses as `application/problem+json` with
  `$ref: '#/components/responses/<BadRequest|NotFound|PreconditionFailed>'`.

## Authoring examples from the contract

Every request body and every success response with content MUST include an
`examples:` block (or `example:` for single shapes). Build examples
directly from the existing types in `Contract.yaml` so they stay in sync.

### Example — request body (`CreateCharacterDto`)

```yaml
post:
    summary: Create a character
    operationId: createCharacter
    requestBody:
        required: true
        content:
            application/json:
                schema:
                    $ref: '#/components/schemas/CreateCharacterDto'
                examples:
                    standard:
                        summary: Minimal valid create
                        value:
                            name: "Gandalf"
                    maxLengthName:
                        summary: Name at the 50-char upper bound
                        value:
                            name: "Gandalf the Grey, Wizard of the Secret Fire"
    responses:
        '201':
            description: Created
            content:
                application/json:
                    schema:
                        type: string
                        format: uuid
                    example: "0b6c1f8a-9c2e-4a18-93a5-3a5e3a5e3a5e"
        '400':
            $ref: '#/components/responses/BadRequest'
```

### Example — success response (`CharacterPayload[]`)

```yaml
responses:
    '200':
        description: List of characters
        content:
            application/json:
                schema:
                    type: array
                    items:
                        $ref: '#/components/schemas/CharacterPayload'
                examples:
                    page:
                        summary: One page of two characters
                        value:
                            -   id: "0b6c1f8a-9c2e-4a18-93a5-3a5e3a5e3a5e"
                                name: "Gandalf"
                                knowCharacterIds:
                                    - "11111111-1111-4111-8111-111111111111"
                                    - "22222222-2222-4222-8222-222222222222"
                            -   id: "11111111-1111-4111-8111-111111111111"
                                name: "Frodo"
                                knowCharacterIds: [ ]
```

### Example — relationship body + path response

```yaml
post:
    operationId: createKnowRelationship
    requestBody:
        required: true
        content:
            application/json:
                schema:
                    $ref: '#/components/schemas/CreateKnowsDto'
                examples:
                    standard:
                        summary: Gandalf knows Frodo
                        value:
                            fromCharacterId: "0b6c1f8a-9c2e-4a18-93a5-3a5e3a5e3a5e"
                            toCharacterId: "11111111-1111-4111-8111-111111111111"
                            description: "Met in the Shire"
```

```yaml
responses:
    '200':
        description: Path found or empty path if no connection
        content:
            application/json:
                schema:
                    $ref: '#/components/schemas/RelationPathPayload'
                examples:
                    twoHopPath:
                        summary: Two-hop path between characters
                        value:
                            characterIds:
                                - "0b6c1f8a-9c2e-4a18-93a5-3a5e3a5e3a5e"
                                - "11111111-1111-4111-8111-111111111111"
                                - "22222222-2222-4222-8222-222222222222"
                            hops: 2
                    noPath:
                        summary: No connection between characters
                        value:
                            characterIds: [ ]
                            hops: 0
```

### Example — error response (`application/problem+json`)

Reuse `$ref: '#/components/responses/<NotFound|BadRequest|PreconditionFailed>'`
instead of inlining the schema. Add example payloads at the **response**
component so every reuse benefits:

```yaml
components:
    responses:
        NotFound:
            description: The specified resource was not found
            content:
                application/problem+json:
                    schema:
                        $ref: '#/components/schemas/ProblemDetails'
                    example:
                        type: "https://tools.ietf.org/html/rfc7231#section-6.5.4"
                        title: "Resource not found"
                        status: 404
                        detail: "Character with id 0b6c1f8a-9c2e-4a18-93a5-3a5e3a5e3a5e not found"
                        instance: "/v1/characters/0b6c1f8a-9c2e-4a18-93a5-3a5e3a5e3a5e"
```

## Adding a new endpoint — checklist

When adding an operation to `Contract.yaml`, verify each item:

- [ ] `operationId` in `camelCase`, unique across the spec.
- [ ] `summary` and (for non-trivial endpoints) `description`.
- [ ] Path parameters typed via reusable `$ref:
      '#/components/parameters/<Name>'` whenever the same parameter
      already exists (e.g. `Id`, `IfMatch`).
- [ ] Request schema referenced from `components/schemas/` — do not inline.
- [ ] Every response status is declared, including error codes:
      `400` → `BadRequest`, `404` → `NotFound`, `412` →
      `PreconditionFailed`, `422` (when validation produces a typed
      error). Always `application/problem+json` for errors.
- [ ] Every body/response has an `examples:` (or `example:`) block whose
      value satisfies the schema constraints.
- [ ] All bounds from the table above are present on every schema you
      touched.
- [ ] After saving, run validation (next section).

## Validation

Run the validators before committing the contract:

```bash
# Spectral (lint)
npx --yes @stoplight/spectral-cli lint backend/Utilities/Contract.yaml

# Redocly (lint + bundle preview)
npx --yes @redocly/cli lint backend/Utilities/Contract.yaml
npx --yes @redocly/cli preview-docs backend/Utilities/Contract.yaml
```

Minimal Spectral ruleset (project-local `.spectral.yaml`):

```yaml
extends: ["spectral:oas"]
rules:
  operation-operationId: error
  operation-description: warn
  operation-success-response: error
  oas3-valid-media-example: error
  oas3-valid-schema-example: error
  # Bounds-required rules — keep these on, they enforce the table above.
  string-bounds:
    description: Free-text strings must declare minLength and maxLength.
    severity: error
    given: "$..properties[?(@.type == 'string' && !@.format && !@.enum)]"
    then:
      - field: minLength
        function: truthy
      - field: maxLength
        function: truthy
  array-bounds:
    description: Arrays must declare minItems and maxItems.
    severity: error
    given: "$..properties[?(@.type == 'array')]"
    then:
      - field: minItems
        function: defined
      - field: maxItems
        function: truthy
  integer-bounds:
    description: Integers and numbers must declare minimum and maximum.
    severity: error
    given: "$..properties[?(@.type == 'integer' || @.type == 'number')]"
    then:
      - field: minimum
        function: defined
      - field: maximum
        function: defined
```

## Regenerating the frontend client

After changing `Contract.yaml`, regenerate
`frontend/src/services/httpClients/RpgAssistantClient.ts` using NSwag.
The file's `<auto-generated>` banner must stay — never hand-edit. After
regeneration:

1. Run `bun run type-check` to detect breaking shape changes.
2. Update `frontend/src/services/RpgAssistantService.ts` to expose any
   new operations and translate the generated DTOs into
   `frontend/src/services/Models/*` classes.
